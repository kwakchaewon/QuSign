# 하네스 엔지니어링 구조 설계서

> AI 에이전트와 함께 작업하는 모든 프로젝트의 **시작점 템플릿**.
> 새 프로젝트를 시작할 때 이 문서를 참조해 폴더 구조와 규칙 파일을 세팅한다.

---

## 1. 설계 철학

### 1.1 진입점은 얇게, 규칙은 분산
- `CLAUDE.md`는 100줄 이내. 라우팅만 담당.
- 세부 규칙은 `harness/` 하위에 영역별로 분산.

### 1.2 자유와 제약의 분리
- **구현 방식**: 자유
- **금지 사항** (`SECURITY.md`): 물리적 잠금
- **검증 통과** (`RELIABILITY.md`, `QUALITY_SCORE.md`): 강제

### 1.3 문서 = 시스템의 본체
- `docs/`는 코드의 부속물이 아니라 **진실의 근원(SSOT)**.
- 변화 속도별로 5개 폴더로 분리.

### 1.4 영역별 차등 품질
- `QUALITY_SCORE.md`로 영역마다 S/A/B/C/D 등급 부여.
- 모든 코드를 같은 기준으로 평가하지 않음 → 과잉 엔지니어링 방지.

### 1.5 점진적 반복 + 자체 검증
- 한 번에 끝내지 않는다. 작은 변경 → 검증 → 다음.
- 매주 금요일 20분, 하네스 자체를 회고·개선.

### 1.6 에이전트 능력의 재사용 (스킬)
- 반복되는 작업 패턴은 **스킬로 추출**해 `.claude/skills/`에 저장.
- 규칙(harness)이 "지켜야 할 것"이라면, 스킬은 "할 수 있는 것"이다.

## 2. 전체 폴더 구조

```
project-root/
│
├── CLAUDE.md                  ← 에이전트 진입점 (~100줄, 라우팅만)
├── ARCHITECTURE.md            ← 시스템 지도 (도메인 맵 + 레이어링)
├── README.md                  ← 사람용 프로젝트 안내
│
├── .claude/                   ← 에이전트 능력 확장
│   └── skills/                ← 재사용 가능한 작업 패턴
│       ├── auto-commit/       ← 프로젝트 커밋 스킬
│       │   └── SKILL.md
│       ├── (예: api-contract-test)/
│       │   └── SKILL.md
│       └── (예: rag-ingestion)/
│           └── SKILL.md
│
├── docs/                      ← 단일 진실 공급원 (SSOT)
│   ├── design-docs/           ← 검증 완료된 결정 (ADR)
│   ├── exec-plans/            ← 진행 중 계획 + 결정 로그
│   ├── generated/             ← 코드에서 자동 생성 (사람 편집 금지)
│   ├── product-specs/         ← 제품 요구사항
│   └── references/            ← 외부 라이브러리 재포맷
│
└── harness/                   ← 규칙 (영역별 분산)
    ├── CHANGELOG.md           ← 하네스 자체의 변경 이력
    ├── SECURITY.md            ← 절대 금지 + 가드레일
    ├── WORKFLOW.md            ← 작업 흐름 + 플랜 양식
    ├── PRODUCT_SENSE.md       ← 비즈니스 맥락
    ├── QUALITY_SCORE.md       ← 영역별 품질 등급
    ├── RELIABILITY.md         ← 타임아웃, 재시도, 장애 복구
    │
    └── (필요 시 추가)
        ├── BACKEND.md
        ├── FRONTEND.md
        ├── RAG.md
        ├── INFRA.md
        └── DOCS.md
```

## 3. 각 파일의 역할

### 3.1 루트 파일

| 파일 | 역할 | 분량 가이드 |
|---|---|---|
| `CLAUDE.md` | 에이전트가 가장 먼저 보는 진입점. "어디로 가야 하는지"만 안내. | ~100줄 |
| `ARCHITECTURE.md` | 시스템의 도메인 맵, 패키지 레이어링, 데이터 흐름. | ~200줄 |
| `README.md` | 사람용. 프로젝트 개요, 설치, 실행 방법. | 자유 |

### 3.2 `.claude/skills/` — 에이전트 능력

스킬은 **반복되는 작업 패턴을 재사용 가능한 형태로 추출한 단위**다.
규칙이 "하지 말 것"이라면, 스킬은 "이렇게 하라"이다.

| 폴더 구조 | 역할 |
|---|---|
| `.claude/skills/<skill-name>/` | 한 스킬당 한 폴더 |
| `.claude/skills/<skill-name>/SKILL.md` | 스킬의 메인 정의 (필수) |
| `.claude/skills/<skill-name>/templates/` | 스킬이 사용하는 템플릿 (선택) |
| `.claude/skills/<skill-name>/examples/` | 사용 예시 (선택) |

**스킬을 만들기 좋은 신호:**
- 같은 작업을 3번 이상 반복했다
- 매번 비슷한 프롬프트를 입력하고 있다
- 일관된 형식의 산출물이 필요하다 (커밋 메시지, 보고서, 리뷰 등)
- 도메인 특화 절차가 있다 (예: RAG 파이프라인 인입, 보안 점검)

**스킬을 만들지 말아야 할 신호:**
- 1~2번만 쓸 일회성 작업
- 매번 맥락이 크게 달라지는 작업
- harness/ 규칙으로 충분히 표현되는 것

### 3.3 `docs/` — 단일 진실 공급원

문서를 **변화 속도별로** 분리한다. 이 분리가 핵심이다.

| 폴더 | 변화 속도 | 진실의 출처 | 무엇을 담나 |
|---|---|---|---|
| `design-docs/` | 거의 안 변함 | 사람의 합의 | ADR (Architecture Decision Record) |
| `exec-plans/` | 작업 단위 | 사람 + 에이전트 | 플랜 + 진행 로그 + 결정 로그 |
| `generated/` | 코드와 함께 | **코드 자체** | OpenAPI, 타입 정의, 의존성 그래프 |
| `product-specs/` | 분기 단위 | 제품/비즈니스 | PRD, 사용자 스토리, 비즈니스 룰 |
| `references/` | 외부 라이브러리 변경 시 | 외부 (재포맷됨) | 자주 쓰는 API의 LLM용 요약본 |

> **`generated/`를 분리하는 이유**: "문서-코드 불일치"가 가장 자주 발생하는 영역인데, **코드에서 자동 생성**하면 불일치 자체가 발생할 수 없다. 검증 비용을 0으로 만든다.

### 3.4 `harness/` — 규칙

| 파일 | 역할 |
|---|---|
| `SECURITY.md` | 절대 금지 사항(NEVER), 명시적 승인 필요 항목(ASK), 화이트리스트, 인젝션 방어 |
| `WORKFLOW.md` | 플랜 양식, 작업 사이클, 변경 단위 가이드, 검증 게이트, 병렬 정책 |
| `PRODUCT_SENSE.md` | 제품 비전, 핵심 사용자, 의사결정 우선순위, 안 하기로 한 것 |
| `QUALITY_SCORE.md` | 영역별 S/A/B/C/D 등급표, 등급별 요구사항 (테스트 커버리지, 모니터링 등) |
| `RELIABILITY.md` | 타임아웃, 재시도, 장애 복구, 모니터링 임계치, 백업 |
| `CHANGELOG.md` | 하네스 자체의 변경 이력 (매주 금요일 회고 시 갱신) |

## 4. 각 파일에 들어갈 내용 (템플릿)

### 4.1 `CLAUDE.md` 골격

```markdown
# CLAUDE.md

## 0. 작업 시작 전 체크리스트
1. 이 파일을 읽는다
2. ARCHITECTURE.md로 시스템 지도 파악
3. §1 절대 금지 확인
4. §4 라우팅에서 작업 유형별 세부 문서 읽기
5. 적용 가능한 스킬이 있는지 .claude/skills/ 확인
6. 플랜 먼저 작성
7. 사용자 승인 후 구현

## 1. 절대 금지 사항 (NEVER)
- ❌ 외부 네트워크 전송 (화이트리스트 외)
- ❌ 비밀 정보 읽기·출력·전송
- ❌ 영구 삭제
- ❌ 권한 변경
- ❌ 결제·금융 거래
- ❌ 사용자 명시 승인 없는 배포

## 2. 자유와 제약의 분리
| 영역 | 정책 |
| 구현 방식 | 자유 |
| §1 금지 | 물리적 잠금 |
| §3 검증 | 강제 |

## 3. 검증
- 즉시: 자동 테스트, 타입 체크, 린터
- 주기: 정리 / 문서-코드 일치 / 아키텍처 위반 검사 에이전트
- 회고: 매주 금요일 20분

## 4. 규칙 라우팅
| 작업 유형 | 참조 문서 |
| 시스템 지도 | ARCHITECTURE.md |
| 보안 / 금지 | harness/SECURITY.md |
| 작업 흐름 | harness/WORKFLOW.md |
| 사용 가능한 스킬 | .claude/skills/ |
| ... | ... |

## 5. 메타 원칙
> 최소한의 규칙으로 최대 효율. 규칙 추가 전 묻는다: "이게 정말 사고를 막는가?"

## 6. 변경 이력
harness/CHANGELOG.md 참조.
```

### 4.2 `ARCHITECTURE.md` 골격

```markdown
# ARCHITECTURE.md

## 1. 한 문장 요약
> (이 시스템이 무엇을 하는지)

## 2. 도메인 맵
| 도메인 | 역할 | 주요 진입점 |

## 3. 패키지 레이어링
[ Presentation ] → [ Application ] → [ Domain ] → [ Infrastructure ]
의존성 방향: 위→아래만 허용

## 4. 외부 의존성
| 시스템 | 용도 | 장애 시 영향 |

## 5. 데이터 흐름
(다이어그램 또는 design-docs 링크)

## 6. 변경 시 영향 범위 매트릭스
| 변경 영역 | 영향 받는 영역 | 필요한 검증 |
```

### 4.5 `harness/WORKFLOW.md` 골격

```markdown
# WORKFLOW.md

## 1. 기본 사이클
플랜 → 승인 → 작은 변경 → 검증 → (반복) → 종료 → 회고

## 2. 플랜 양식
[목표] [변경 대상] [단계] [검증 방법] [리스크] [중단 기준]
- 적용 스킬: .claude/skills/ 중 사용할 스킬 명시

## 3. 변경 단위 가이드
- 파일 수: 1~3개
- 코드 라인: 100줄 이내
- 도메인 경계: 절대 동시에 넘지 않음

## 4. 검증 게이트
[ ] 자동 테스트
[ ] 타입 체크
[ ] 린터
[ ] generated 동기화

## 5. 병렬 작업 정책
- 독립 도메인일 때만
- 검증 큐가 밀리면 즉시 ↓

## 6. 작업 종료 시
- exec-plans에 결정 로그
- 코드-문서 동기화 확인
- 반복된 패턴 발견 시 → .claude/skills/로 추출 검토

## 7. 회고 (매주 금요일 20분)
- 새로 만들면 좋을 스킬은?
- 기존 스킬 중 사용 안 된 것은?
```

```markdown
# QUALITY_SCORE.md

## 1. 등급 체계
| 등급 | 기준 | 적용 영역 |

| 항목 | S | A | B | C | D |

| 통합 테스트 | 필수 | 필수 | 권장 | - | - |
| 타입 체크 | strict | strict | normal | normal | - |
| 모니터링 | 필수 | 필수 | 권장 | - | - |
| 롤백 절차 | 필수 | 필수 | 권장 | - | - |

## 3. 본 프로젝트의 영역별 등급
| 영역 | 등급 | 사유 |

## 4. 등급 변경 절차

## 5. 적용 시 주의
- S/A 영역: 자율 에이전트의 자동 변경 금지
- D 영역: main 직접 머지 금지
```

### 4.8 `harness/RELIABILITY.md` 골격

```markdown
# RELIABILITY.md

## 1. 타임아웃 기본값
| 외부 API | 30초 |
| LLM 추론 | 60초 |
| DB 쿼리 | 5초 |
| 벡터 검색 | 3초 |

| 5xx | 3회, 지수 백오프 |
| 429 | 5회, retry-after 준수 |
| 4xx | 재시도 안 함 |
| 비멱등 | 재시도 안 함 |

## 3. 장애 시 동작
- Graceful Degradation
- Circuit Breaker
- 데이터 무결성 보호

## 4. 모니터링 임계치
| 에러율 > 1% | 즉시 |
| P95 > 3초 | 5분 후 |
| LLM 비용 > 80% | 즉시 |

## 5. 백업

## 6. 등급별 차등 (QUALITY_SCORE 연계)
| S | 99.9%+ | RTO < 15분 |
| A | 99%+ | RTO < 1시간 |
| B | 95%+ | RTO < 4시간 |
```

```markdown
- 어떤 규칙/스킬을 왜 추가했나

### 변경
- 어떤 규칙/스킬을 어떻게 바꿨나, 왜

### 삭제
- 어떤 규칙/스킬을 왜 뺐나

### 회고
- 지난주 어떤 규칙이 작동했나
- 어떤 스킬이 자주 사용됐나 / 사용 안 됐나
- 어떤 규칙이 무시·우회됐나
- 다음주 시도할 것
```

## 5. 스킬 작성 가이드

### 5.1 스킬의 구조

```
.claude/skills/<skill-name>/
├── SKILL.md           ← 필수. 스킬의 메인 정의
├── templates/         ← 선택. 템플릿 파일
│   └── *.md, *.json
└── examples/          ← 선택. 사용 예시
    └── *.md
```

### 5.2 SKILL.md의 필수 요소

| 요소 | 역할 |
|---|---|
| frontmatter `name` | 스킬의 고유 식별자 |
| frontmatter `description` | 트리거 조건. 에이전트가 이 스킬을 언제 호출할지 판단하는 근거 |
| "언제 사용하나" | 구체적 트리거 상황 |
| "언제 사용하지 않나" | 혼동을 방지하는 음의 정의 |
| 절차 | 따를 단계 |

> **description이 가장 중요하다.** 에이전트는 description을 보고 스킬을 선택한다. 모호한 description은 호출되지 않거나 잘못 호출된다.

### 5.3 좋은 스킬의 예시

| 스킬 | 트리거 | 산출물 |
|---|---|---|
| `auto-commit` | 커밋 메시지 작성 요청 | Conventional Commits 형식의 메시지 |
| `adr-write` | "이 결정을 기록하자" | docs/design-docs/ADR-NNN.md |
| `rag-ingestion` | 새 문서 RAG 인입 요청 | Qdrant에 임베딩 + 메타데이터 |
| `security-incident-report` | 보안 사고 발생 | 내부용 + 고객용 보고서 2종 |
| `quarterly-investment-review` | 분기 회고 시점 | 4-Factor 체크리스트 결과 |

### 5.4 스킬과 harness/ 규칙의 관계

| 측면 | harness/ 규칙 | .claude/skills/ |
|---|---|---|
| 성격 | 제약 (negative) | 능력 (positive) |
| 적용 | 항상 적용 | 트리거될 때만 |
| 위반 시 | 작업 중단 | 그냥 사용 안 됨 |
| 변경 빈도 | 낮음 | 중간 |
| 예시 | "비밀번호 출력 금지" | "커밋 메시지 자동 생성" |

> 스킬도 harness/ 규칙을 지켜야 한다. 스킬이 §1 금지 사항을 우회하지 않는지 매주 회고에서 점검.

## 6. 프로젝트 규모별 적용 가이드

모든 파일을 다 채울 필요 없다. **규모에 맞게 켜고 끈다.**

### S — 1인 사이드 프로젝트
**예: AdLex, 개인 도구**

필수 파일:
- `CLAUDE.md`
- `ARCHITECTURE.md`
- `harness/SECURITY.md`
- `harness/WORKFLOW.md`
- `harness/CHANGELOG.md`
- `.claude/skills/auto-commit/` (반복 작업 1개부터)

선택 (나중에 추가):
- 나머지 모두

### M — 연구실 / 소규모 팀 프로젝트
**예: Akashic Records**

위 + 추가:
- `harness/RELIABILITY.md`
- `harness/QUALITY_SCORE.md`
- `docs/design-docs/`
- `docs/exec-plans/`
- `.claude/skills/` 3~5개 (도메인 특화 스킬)

### L — 다중 소스 / 운영 시스템
**예: ProjectMind v3.0, Worktronics**

전체 사용. 영역별 추가 가이드 작성:
- `harness/BACKEND.md`
- `harness/FRONTEND.md`
- `harness/RAG.md`
- `harness/INFRA.md`
- `harness/DOCS.md`
- `.claude/skills/` 5~10개 (전체 워크플로우 커버)

## 7. 새 프로젝트 시작 절차

1. 프로젝트 루트에 위 폴더 구조 생성
2. `CLAUDE.md` §1 금지 항목과 §4 라우팅을 프로젝트에 맞게 수정
3. `ARCHITECTURE.md` §1 한 문장 요약과 §2 도메인 맵 작성
4. `harness/SECURITY.md` 화이트리스트 도메인 갱신
5. `harness/PRODUCT_SENSE.md` §1~§4 작성
6. **`.claude/skills/`에 가장 자주 반복할 작업 1개를 스킬로 추출**
7. 규모(S/M/L)에 맞춰 불필요한 파일 삭제 또는 비워둠
8. 첫 작업 플랜을 `docs/exec-plans/`에 작성 후 시작

## 8. 두 접근의 통합 — 무엇을 어디서 가져왔나

### 채원님의 원칙에서 가져온 것
- 얇은 진입점 (CLAUDE.md 100줄 이내)
- §1 절대 금지 / §2 자유와 제약 / §3 검증 / §4 라우팅 골격
- 매주 금요일 20분 회고
- 메타 원칙: 최소한의 규칙으로 최대 효율
- **반복 작업 하나를 골라 규칙(또는 스킬)로 만든다**

### 외부 패턴(이미지 사례)에서 가져온 것
- `docs/`를 SSOT로 격상 (변화 속도별 5분류)
- `generated/` 분리로 문서-코드 불일치 구조적 차단
- `PRODUCT_SENSE.md`로 비즈니스 맥락을 명시적 입력으로
- `QUALITY_SCORE.md`로 영역별 차등 품질
- `RELIABILITY.md`로 신뢰성 기준 분리

### 통합하면서 추가한 것
- 프로젝트 규모별 적용 가이드 (S/M/L)
- 품질 등급(S/A/B/C/D)과 신뢰성 목표(가용성, RTO) 연계
- 각 폴더의 "여기 넣지 말 것" 명시
- **`.claude/skills/`로 에이전트의 능력을 재사용 가능하게 추출**
- **harness(제약) ↔ skills(능력)의 이중 구조**

## 9. 메타 원칙 (재강조)

> **최소한의 규칙으로 최대 효율을 낸다.**
>
> 규칙을 추가할 때마다 묻는다: "이게 정말 사고를 막는가?"
>
> - 답이 모호하면 → 추가하지 않는다
> - 답이 분명하면 → `harness/SECURITY.md`(금지) 또는 `harness/RELIABILITY.md`(검증)에 편입
>
> 스킬을 추가할 때마다 묻는다: "이 작업을 3번 이상 반복했는가?"
>
> - 답이 No → 추가하지 않는다
> - 답이 Yes → `.claude/skills/`로 추출
>
> 규칙은 "어떻게 할지"가 아니라 "**무엇을 지켜야 할지**"에만 건다.
> 스킬은 "**어떻게 잘할지**"의 모범 답안을 박제한다.
